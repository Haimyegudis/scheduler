# תכנון: מערכת שיבוץ משמרות לטכנאים

**תאריך:** 2026-07-21 (עודכן: מודל הרשאות חדש — כניסה במייל, רשימת מיילים מורשים)
**סטטוס:** מאושר על ידי המשתמש

## מטרה

אפליקציית Web לניהול אילוצים ושיבוץ משמרות שבועי לטכנאים. שני סוגי משתמשים: טכנאי ומנהל. המנהל מייצר תוכנית שיבוץ אוטומטית לפי אילוצים, עורך ידנית, ומפרסם.

## החלטות מפתח (מאישור המשתמש)

- **פריסה:** אינטרנט, חינמי. Next.js (App Router) על Vercel + Neon Postgres (free tier).
- **גודל צוות:** 13–20 טכנאים.
- **ימי עבודה:** ראשון–חמישי. שישי אופציונלי — המנהל מדליק לשבוע ספציפי. שבת לא מאוישת.
- **מחסור בטכנאים:** עמדות נשארות ריקות ומסומנות באדום; המנהל פותר ידנית. אין שיבוץ בניגוד לאילוצים אוטומטית.
- **טכנאים רואים את התוכנית שפורסמה** — הטבלה המלאה, משמרות שלהם מודגשות.
- **עמדות זהות** — "עמדה 1–4", כל טכנאי מתאים לכל עמדה.
- **שפה:** עברית, RTL, רספונסיבי למובייל.

### מודל הרשאות (עדכון 2026-07-21)

- **אין כניסת admin/admin123.** כולם נכנסים עם מייל + סיסמה.
- **הרשמה רק למיילים מורשים:** המנהל מזין רשימת מיילים מורשים. מייל שאינו ברשימה — ההרשמה נדחית עם הודעה "המייל אינו מורשה, פנה למנהל".
- **מנהל ראשון (bootstrap):** המייל `gorgani@hp.com` מורשה מראש (קבוע בקוד); בהרשמה הראשונה עם מייל זה המשתמש מקבל הרשאות מנהל אוטומטית.
- **הענקת ניהול:** מנהל יכול להעניק/להסיר הרשאת מנהל לכל משתמש אחר. מנהל לא יכול להסיר את ההרשאה של עצמו.
- **מנהל אינו טכנאי:** משתמש עם הרשאת מנהל לא מופיע ברשימות הטכנאים, לא ממלא אילוצים ולא משובץ למשמרות.
- הרשאה שהוענקה/הוסרה נכנסת לתוקף בכניסה הבאה של אותו משתמש (ההרשאה נשמרת בטוקן).

## ארכיטקטורה

- **Next.js** אפליקציה אחת: React frontend + API route handlers.
- **DB:** Postgres (Neon בענן; SQLite מקומי לפיתוח). ORM: Prisma.
- **Auth:** JWT ב-httpOnly cookie (Secure ב-production). הרשמה/התחברות במייל+סיסמה (bcrypt hash, מינימום 8 תווים, ללא אימות מייל). תפקיד בטוקן: `admin` אם isAdmin, אחרת `technician`.
- **זיהוי שבוע:** לפי תאריך יום ראשון של השבוע (`weekStart`, ISO date). ניווט חצים = ±7 ימים.

## מודל נתונים

### Technician (כל המשתמשים, כולל מנהלים)
- `id`, `name`, `email` (unique, נשמר באותיות קטנות), `passwordHash`, `isAdmin` (boolean, ברירת מחדל false), `createdAt`

### AllowedEmail (רשימת מיילים מורשים להרשמה)
- `id`, `email` (unique, אותיות קטנות), `createdAt`
- המייל `gorgani@hp.com` מורשה תמיד (קבוע `ADMIN_EMAIL` בקוד, לא חייב שורה בטבלה).

### Constraint
- `id`, `technicianId`, `date` (ISO date), `value`: `morning | evening | flex | off`
- unique על (`technicianId`, `date`)

### Schedule (תוכנית שבועית)
- `id`, `weekStart` (unique), `status`: `draft | published`, `includeFriday` (boolean), `createdAt`, `updatedAt`

### Assignment
- `id`, `scheduleId`, `date`, `shift`: `morning | evening`, `station`: 1–4, `technicianId`
- unique על (`scheduleId`, `date`, `shift`, `station`)

## מסכים

### כולם
1. **הרשמה** — שם, מייל, סיסמה (8+). רק מייל מורשה; אחרת שגיאה ברורה. מייל תפוס → שגיאה.
2. **התחברות** — מייל + סיסמה. הפניה אוטומטית: מנהל → לוח בקרה, טכנאי → אילוצים.

### טכנאי
3. **אילוצים שבועיים** — חצים לניווט בין שבועות. ימים ראשון–חמישי (+שישי אם הודלק). לכל יום 4 כפתורים: בוקר/ערב/גמיש/חופש. ברירת מחדל: לא נבחר ("לא מילא"). שמירה אוטומטית. אחרי פרסום — נעול + הסבר.
4. **צפייה בתוכנית** — אם פורסמה: טבלה מלאה, המשמרות שלו מודגשות. אחרת: "טרם פורסמה תוכנית".

### מנהל
5. **לוח בקרה (לשבוע נבחר, חצים)** — רשימת טכנאים (ללא מנהלים): שם + סטטוס מילוי (מלא/חלקי/לא מילא) + טבלת אילוצים (שורות=טכנאים, עמודות=ימים, צבע לכל סוג).
6. **תוכנית משמרות** — מתג "כולל שישי" (נשמר מיידית, גם לפני יצירת תוכנית — כך טכנאים רואים שישי מראש). כפתור "צור תוכנית" (דורס טיוטה עם אישור). טבלה: ימים × (בוקר: עמדות 1–4, ערב: עמדות 1–4), כל תא dropdown; שיבוץ לא-זמין או כפול-ביום מותר עם אזהרה. עמדה ריקה = אדום. שמור טיוטה / פרסם (גם פרסום חוזר). שורת סיכום משמרות לטכנאי.
7. **ניהול משתמשים** —
   - רשימת מיילים מורשים: הוספה, הסרה.
   - רשימת משתמשים רשומים: שם, מייל, מתג "מנהל" (הענקה/הסרה). המתג של המנהל המחובר עצמו נעול.

## אלגוריתם השיבוץ

קלט: `weekStart`, `includeFriday`, אילוצי כל הטכנאים (isAdmin=false בלבד) לשבוע.

לכל יום (ראשון–חמישי, +שישי אם פעיל), לכל משמרת בסדר בוקר→ערב:

1. **זמינים:** אילוץ = ערך המשמרת או `flex`. `off` או חסר → לא זמין. מי שכבר שובץ באותו יום — נפסל.
2. **מילוי 4 עמדות:** בכל פעם הזמין עם מספר המשמרות המצטבר הנמוך ביותר בשבוע (שוויון → פחות משמרות מהסוג הזה, אחר כך id). בבוקר, בשוויון — עדיפות ל-`morning` על `flex` (לשמור גמישים לערב).
3. **מחסור:** פחות מ-4 זמינים → עמדות נותרות ריקות (אין רשומת Assignment).

חוקים קשיחים שהאלגוריתם לעולם לא מפר: טכנאי אחד לכל עמדה-משמרת; אין טכנאי פעמיים באותו יום; שיבוץ רק לפי אילוצים. עריכה ידנית של המנהל רשאית לחרוג — עם אזהרה ב-UI בלבד.

## API (route handlers)

- `POST /api/auth/register` — name, email, password → נדחה אם המייל לא מורשה (403) או תפוס (409); `gorgani@hp.com` → isAdmin=true. מחזיר cookie.
- `POST /api/auth/login` — email, password → cookie (role לפי isAdmin).
- `POST /api/auth/logout`
- `GET /api/constraints?weekStart=` — אילוצי הטכנאי המחובר (טכנאי בלבד).
- `PUT /api/constraints` — עדכון אילוץ { date, value }. נדחה אם התוכנית פורסמה (409).
- `GET /api/schedule?weekStart=` — לטכנאי: רק אם published. למנהל: תמיד. רשימת טכנאים ללא מנהלים.
- **מנהל בלבד (403 לאחרים):**
  - `GET /api/admin/overview?weekStart=` — טכנאים (ללא מנהלים) + סטטוסים + טבלת אילוצים.
  - `POST /api/admin/schedule/generate` — { weekStart, includeFriday } → יצירת/דריסת טיוטה.
  - `PUT /api/admin/schedule` — עדכון שיבוצים ידני + includeFriday.
  - `POST /api/admin/schedule/publish` — { weekStart } → published.
  - `GET/POST/DELETE /api/admin/allowed-emails` — רשימה / הוספה { email } / הסרה { email }.
  - `GET /api/admin/users` — כל המשתמשים: id, name, email, isAdmin.
  - `PUT /api/admin/users` — { userId, isAdmin } → עדכון הרשאה. שינוי עצמי → 400.

## טיפול בשגיאות

- מייל לא מורשה בהרשמה → 403 + "המייל אינו מורשה, פנה למנהל".
- מייל תפוס → 409. סיסמה קצרה → 400 (ולידציה גם בלקוח).
- עריכת אילוץ לשבוע שפורסם → 409 + נעילת UI.
- גישה ללא הרשאה → 401/403.
- מנהל מנסה להסיר את ההרשאה של עצמו → 400.
- אין מספיק זמינים → לא שגיאה; תוצאה חלקית + סימון אדום.
